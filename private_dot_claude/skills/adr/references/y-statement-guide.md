# Y-Statement Guide

Y-Statementは、アーキテクチャ決定の本質を1文で表現する形式。ADRのフロントマターに `y-statement` フィールドとして記載し、意思決定の要約として機能する。

## 短縮形（Short Form）

```
<ユースケース>の文脈において、
<懸念事項>に直面したため、
<品質目標>を達成するために、
<欠点>を受け入れ、
<選択肢>を採用することを決定した。
```

### 構成要素

| 要素 | 説明 | 例 |
| --- | --- | --- |
| ユースケース | どのような状況か | 「ユーザー認証機能の実装」 |
| 懸念事項 | 何が問題か | 「セッション管理の複雑さとセキュリティリスク」 |
| 品質目標 | 何を達成したいか | 「ステートレスな認証とスケーラビリティ」 |
| 欠点 | 受け入れるトレードオフ | 「トークン失効の即時性が犠牲になること」 |
| 選択肢 | 何を採用するか | 「JWTベースの認証」 |

### 例

#### 例1: 認証方式

```yaml
y-statement: >
  ユーザー認証機能の実装において、
  セッション管理の複雑さとセキュリティリスクに直面したため、
  ステートレスな認証とスケーラビリティを達成するために、
  トークン失効の即時性が犠牲になることを受け入れ、
  JWTベースの認証を採用することを決定した。
```

#### 例2: データベース選択

```yaml
y-statement: >
  リアルタイム通知機能のデータストアにおいて、
  高頻度の書き込みと低レイテンシの読み取り要件に直面したため、
  ミリ秒単位の応答性能を達成するために、
  データの永続性保証が弱くなることを受け入れ、
  Redisをプライマリストアとして採用することを決定した。
```

#### 例3: テストフレームワーク

```yaml
y-statement: >
  フロントエンドのユニットテスト環境の構築において、
  既存のJestベースのテストが遅くDXが低下している問題に直面したため、
  テスト実行速度の大幅な改善を達成するために、
  Jestエコシステムのプラグイン互換性を失うことを受け入れ、
  Vitestへの移行を採用することを決定した。
```

## 拡張形（Long Form）

短縮形に「却下した選択肢」と「追加の根拠」を付加する形式。

```
<ユースケース>の文脈において、
<懸念事項>に直面したため、
<品質目標>を達成するために、
<欠点>を受け入れ、
<他の選択肢>を却下し、
<選択肢>を採用することを決定した。
なぜなら<追加の根拠>だからである。
```

### 例（拡張形）

```yaml
y-statement: >
  ユーザー認証機能の実装において、
  セッション管理の複雑さとセキュリティリスクに直面したため、
  ステートレスな認証とスケーラビリティを達成するために、
  トークン失効の即時性が犠牲になることを受け入れ、
  サーバーサイドセッションとOAuth 2.0のみの構成を却下し、
  JWTベースの認証を採用することを決定した。
  なぜなら、マイクロサービス間の認証伝搬が最もシンプルになるからである。
```

## フロントマターでの使用

ADRのYAMLフロントマターに `y-statement` フィールドとして記載する:

```yaml
---
status: proposed
date: 2026-03-08
decision-makers: [hayakawa]
y-statement: >
  <Y-Statementをここに記載>
---
```

Y-Statementは短縮形を基本とし、必要に応じて拡張形を使用する。
