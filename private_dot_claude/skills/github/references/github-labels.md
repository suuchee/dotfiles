# GitHub Issue ラベル

Issue 発行時の標準ラベル定義。

## 前提条件

- ラベル付けの前に、リポジトリの性質（アプリケーション / インフラ / ライブラリ等）を把握すること
- プロジェクト固有の CLAUDE.md で判断基準やラベルをオーバーライド可能
- 非機能要件として定義済みの項目は bug / enhancement を適用し、要件が明確でない領域はラベルなしまたはプロジェクトごとに判断
  - 例: アプリケーションリポジトリ内のインフラコード（CI/CD、Dockerfile等）の改善は、非機能要件として定義されていなければ標準ラベルに該当しないため、ラベルなしまたはカスタムラベルで対応

## ラベル一覧

| label | 対象 | 例 | 使用する条件 |
| --- | --- | --- | --- |
| `bug` | 仕様に反する不具合 | 動作不良、エラー、クラッシュ、仕様と異なる表示 | ビジネスロジック・仕様を把握した上で、仕様通りに動作しない問題を報告するとき |
| `enhancement` | 新機能・改善 | 新API、UX向上、機能拡張、デザインリニューアル | 新しい機能や既存機能の改善を提案するとき（ビジネス的な意図を持つUI改善を含む） |
| `documentation` | ドキュメントのみ | README更新、API仕様追加 | コードを伴わないドキュメント変更のとき |
| `question` | 質問・相談 | 使い方、設計意図の確認 | 回答を求める問い合わせのとき |
| `help wanted` | 協力募集 | 実装支援、レビュー依頼 | 他者の協力が必要なとき |
| `good first issue` | 初心者向け | 小規模修正、明確なタスク | コントリビューター入門に適したとき |
| `duplicate` | 重複 | 既存Issueと同一内容 | 同じ問題が既に報告されているとき |
| `invalid` | 無効 | 再現不可、仕様通り | Issue として成立しないとき |
| `wontfix` | 対応しない | 範囲外、設計上の意図 | 対応しないと判断したとき |

## コミットプレフィックスとの対応

Issue ラベルとコミットプレフィックス（git スキル参照）の対応関係:

| Issue ラベル | 主な対応コミットプレフィックス |
| --- | --- |
| `bug` | `fix` |
| `enhancement` | `feat`（ビジネス的意図を持つUI改善も含む） |
| `documentation` | `docs` |

**補足**: 純粋な視覚調整（色変更、余白調整など）は Issue 化されることが少ないため、専用ラベルは設けない。Issue として報告される UI 改善はビジネス的意図を伴うことが多いため `enhancement` を使用する
